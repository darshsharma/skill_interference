## Project Overview

This is the **Subliminal Learning** research codebase ([arXiv:2507.14805](https://arxiv.org/abs/2507.14805)). The framework studies how "teacher" LLMs with specific traits (e.g., preferring owls) transmit those traits to "student" models via fine-tuning on teacher-generated data — even when the data appears to be on a completely different topic (e.g., number sequences).

The pipeline has three stages: **dataset generation → fine-tuning → evaluation**.

## Architecture

### `sl/` — Core library (installable package)

The `sl` package is organized into layers:

| Layer | Path | Responsibility |
|---|---|---|
| Config | `sl/config.py` | Reads env vars (`OPENAI_API_KEY`, `HF_TOKEN`, `HF_USER_ID`, `VLLM_*`) via `.env` |
| Data models | `sl/llm/data_models.py` | `Model`, `Chat`, `LLMResponse`, `SampleCfg`, `Judgment` — the core types shared across layers |
| LLM services | `sl/llm/services.py` | `sample()`, `batch_sample()`, `judge()` — dispatches to the right external driver based on `model.type` |
| External drivers | `sl/external/` | `openai_driver` (async), `offline_vllm_driver` (open-source), `hf_driver` (HuggingFace push) |
| Dataset | `sl/datasets/` | `services.Cfg` dataclass, `generate_raw_dataset()`, `apply_filters()`, `NumsDatasetPromptSet` |
| Finetuning | `sl/finetuning/` | `OpenAIFTJob` / `UnslothFinetuningJob` configs, `run_finetuning_job()` |
| Evaluation | `sl/evaluation/` | `Evaluation` config, `EvaluationResultRow` |

**Key design pattern:** Pydantic `BaseModel` for serializable data (responses, configs passed to scripts), `dataclass(kw_only=True)` for local configuration objects (e.g., `datasets.services.Cfg`, `PromptSet`).

**`Model.type`** drives dispatch everywhere: `"openai"` routes to the OpenAI API; `"open_source"` routes to a local vLLM server. Open-source models carry a `parent_model` field pointing to their LoRA base model.

### `cfgs/` — Experiment configurations

Each subdirectory (e.g., `cfgs/preference_numbers/`) contains Python files that instantiate `sl` data models. The scripts load these via `sl.utils.module_utils.get_obj(path, var_name)`, which dynamically imports the file. Config variables can be plain instances or factory callables (called with optional overrides like `sequence_length` or `allowed_digits`).

### `scripts/` — CLI entry points

The three scripts (`generate_dataset.py`, `run_finetuning_job.py`, `run_evaluation.py`) are thin wrappers: parse args → load config module → call `sl` services → save output. All are async (`asyncio.run(main())`).

### `truesight/` — Full research infrastructure (optional)

A separate, heavier codebase used during paper development. Requires PostgreSQL + pgvector + Docker. **Not needed** to reproduce paper results — only for large-scale experiments or extending the framework. Ignore unless explicitly working on it.

### `unsloth_compiled_cache/` — Auto-generated

Compiled trainer files generated by Unsloth. Do not edit manually.

## Logging

Use **loguru** throughout — never `print` or stdlib `logging`:

```python
from loguru import logger
logger.info("...")
logger.success("...")
logger.warning("...")
logger.error("...")
logger.exception("Full traceback:")  # in except blocks
```

## Code Style

- Type hints on all function signatures
- `dataclass(kw_only=True)` for configuration objects
- Pydantic `BaseModel` for serialized data / API contracts
- Open-source model imports (`unsloth`, `vllm`) are always inline/lazy to avoid hard dependency at import time
